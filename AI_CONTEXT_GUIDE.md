# ProjectHub / AI Claw 新对话预设提示词

你是这个项目的开发 AI。开始新对话时，不要只理解“日志”，而要先理解整个项目、开发约束、数据库机制、日志规范和排障方式。

## 你的任务定位

这个项目是一个面向个人工作的综合平台，包含：

- 项目管理
- 任务管理
- AI 助手
- 资源管理
- 本地代理
- 数据可视化与过程记录

你需要把它当作一个持续演进的完整系统，而不是单纯的前后端 CRUD。

## 你必须先记住的关键规则

### 1. 数据库结构会在后端启动时自动同步

这是本项目非常重要的机制。

- 后端启动时会检查表和列是否存在
- 缺失就自动创建或补齐
- 所以 AI 在新增表、字段、模型时，不能只改实体类或接口
- 必须同步检查 `api/Program.cs` 中的数据库同步逻辑

换句话说，**数据库结构更新不是只靠迁移文件，而是和启动同步机制绑定的。**

### 2. 日志是排障的第一手资料

测试或开发中出现问题时，优先看日志，不要盲猜。

- 前端日志用于记录交互、状态变化、异常
- 后端日志用于记录接口、业务流程、数据库同步、AI、资源访问
- 发现问题后先查日志，再修改代码

### 3. 前端日志不只是接口调用

任何需要交互的地方都建议记录日志，例如：

- 点击按钮
- 打开/关闭弹窗
- 表单提交
- 列表筛选
- Tab / 菜单切换
- 状态流转
- 资源浏览
- AI 对话发送/接收
- 请求成功/失败
- 异常捕获

## 日志系统在哪里看

### 后端日志位置

- 开发时的文件日志通常在 `api/logs/`
- 文件名类似 `app-YYYY-MM-DD.log`
- 如果后端启动失败或数据库同步出错，优先看这里
- 另外也要关注后端启动控制台输出，尤其是数据库同步和 Swagger 相关信息

### 前端日志位置

- 浏览器开发者工具的 Console 面板
- 网络请求失败可在 Network 面板查看
- 如果前端日志已经上报到后端，也可以回到 `api/logs/` 里继续查

## 日志系统怎么记录

### 前端记录方式

前端可以在这些地方记录：

```js
console.log('[TaskList] 点击新增任务')
console.info('[AI] 发送消息', { conversationId })
console.warn('[Resource] 路径为空')
console.error('[Project] 保存失败', error)
```

如果是业务代码，建议记录：

- 当前模块名
- 触发动作
- 关键参数
- 成功/失败结果
- 错误信息

### 后端记录方式

后端建议使用标准日志接口：

```csharp
_logger.LogInformation("创建任务: {TaskId}, {Title}", task.Id, task.Title);
_logger.LogWarning("资源路径不存在: {Path}", path);
_logger.LogError(ex, "AI 请求失败");
```

如果是写入文件日志，可以通过项目里的文件日志服务记录。

### 记录日志的建议格式

建议每条日志都尽量包含：

- 模块名
- 动作
- 关联 ID
- 结果状态
- 错误原因

例如：

- `[Task] 创建任务成功 taskId=12`
- `[Project] 更新失败 projectId=3 reason=validation`
- `[AI] 发送对话失败 conversationId=8`
- `[DB] 同步表结构完成`

## 你在新对话里应该怎么工作

当接到新的开发任务时，先按顺序理解：

1. 项目整体定位
2. 当前技术栈
3. 数据库自动同步机制
4. 日志系统和排障方式
5. 当前任务涉及的控制器、模型、前端页面、服务

不要一上来就大范围翻代码。

## 开发优先级

1. 保持当前架构一致性
2. 保持数据库同步机制可运行
3. 保持日志足够可追踪
4. 再做功能开发或重构

## 生产与开发的区别

- **前端开发日志**：只在开发环境启用
- **后端调试型日志**：尽量只在开发环境使用
- **数据库自动同步**：部署启动时保留，这是项目运行机制的一部分，不要误删

## 你要避免的错误

- 只看日志，不理解项目结构
- 只改模型，不改启动同步逻辑
- 只在接口层记录日志，忽略交互层
- 将开发日志污染到生产环境
- 看到问题就直接改，不先查日志

## 一句话总结

你要把这个项目理解成一个“带数据库自动同步和强排障能力的个人工作平台”，而不是一个普通 CRUD 应用。
