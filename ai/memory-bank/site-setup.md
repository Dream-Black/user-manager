# ProjectHub / AI Claw 项目记忆

## 项目定位

ProjectHub 是一个面向个人工作流的项目管理与资源管理平台，当前主要包含：

- 项目管理
- 任务管理
- AI 助手
- 资源管理
- 本地代理访问

## 技术栈

- **前端**：Vue 3 + Vite + Pinia + TDesign
- **后端**：ASP.NET Core 9 + EF Core + MySQL
- **本地代理**：Python FastAPI
- **部署**：Docker / Docker Compose

## 关键架构规则

### 1. 后端启动会自动同步数据库结构

这是非常重要的规则。

后端在 `api/Program.cs` 里执行数据库同步逻辑，启动时会：

- 检查表是否存在
- 检查列是否存在
- 创建缺失的表
- 补齐缺失的列
- 记录迁移状态

因此在业务开发中，如果 AI 新增模型、字段或表，必须同时检查 `api/Program.cs` 的同步逻辑，不能只改实体类或控制器。

开发时要特别注意：

- 新建模型后，确认启动同步是否会创建对应表
- 新增字段后，确认启动同步是否会补列
- 不要假设只改 `Models` 就够了
- 如果涉及数据库结构变化，优先检查 `Program.cs` 里的初始化逻辑

### 2. 日志系统是 AI 的排查基础

日志主要是给 AI 开发和排障使用。

开发规则：

- 后端关键流程必须写日志
- 前端关键 API 调用、错误处理、状态切换也要写日志
- 测试发现问题后，AI 应优先查日志定位问题
- 不要只凭感觉修改，必须先看日志

建议记录的内容：

- API 请求与响应异常
- 数据库初始化/同步结果
- 任务状态变化
- AI 请求和失败原因
- 文件扫描、漫画读取、资源路径访问异常
- 前端接口失败与页面异常

### 3. 文档分层原则

仓库里的文档分两类：

- **公开文档**：给 Git 访问者/人类看的，例如 `README.md`
- **AI 记忆文档**：给 AI 持续理解项目用的，例如 `ai/memory-bank/` 下的文档

以后更新时要保证：

- 公开文档简洁、面向使用者
- AI 文档强调约束、机制、排障规则
- 不要把所有细节堆在 README 里

## 约定提醒

如果后续 AI 进入新对话，需要先快速读取：

- `README.md`
